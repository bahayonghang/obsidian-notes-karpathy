# Karpathy 工作流

理解 LLM 驱动知识库背后的理念。

## 核心洞见

来自 [Andrej Karpathy 的推文线程](https://x.com/karpathy/status/2039805659525644595)：

> 来自各种来源（推文、文章、论文、视频、代码仓库、播客）的原始数据 → LLM 编译的 `.md` wiki → Q&A、报告、幻灯片——全部可在 Obsidian 中查看。

关键洞见：**将知识管理视为软件编译过程**。

## 传统方法 vs. Karpathy 方法

### 传统知识管理

```
人类阅读来源 → 人类编写 wiki → 人类维护 wiki
```

**问题：**
- 受限于人类的阅读/写作速度
- 质量和格式不一致
- 知识增长后难以维护
- 难以将论断追溯回来源

### Karpathy 方法

```
人类收集来源 → LLM 编译 wiki → LLM 维护 wiki → 人类在 Obsidian 中查询
```

**优势：**
- 人类专注于**策展**（收集什么）
- LLM 处理**编译**（结构化、链接、摘要）
- Wiki 是**确定性的**和**增量的**（像编译器一样）
- 完整的**可追溯性**——从论断到来源

## 编译隐喻

将知识库想象成一个软件项目：

| 软件开发 | 知识库 |
|---------------------|----------------|
| 源代码（`.c`、`.py`） | 原始来源（`raw/*.md`） |
| 编译器（`gcc`、`python`） | LLM 编译（`kb-compile`） |
| 编译后的二进制 | Wiki（`wiki/`） |
| 调试器 | 健康检查（lint） |
| 用户手册 | 报告、幻灯片、图表 |

### 为什么有效

1. **关注点分离**：人类擅长判断什么值得收集；LLM 擅长结构化和摘要
2. **增量更新**：像编译器一样，只处理变更的内容
3. **确定性输出**：相同来源 → 相同 wiki（确定性，非创造性）
4. **完整可追溯性**：每个论断都可追溯到 `[[raw/source]]`

## Wikilink 的力量

Obsidian 的 `[[wikilinks]]` 是秘密武器。它们创建了一个**知识图谱**：

- **概念链接到来源**：每个论断都可追溯
- **来源链接到概念**：每个来源都有贡献
- **概念链接到概念**：想法在整个 wiki 中互相关联

当你提出问题时，LLM 会：
1. 阅读索引文件以了解 wiki 结构
2. 沿着 wikilink 找到相关概念文章
3. 综合多个来源的信息
4. 返回带有完整引用的答案

**不需要 RAG**——wikilink 本身就是检索机制。

## 何时使用这个方法

### 适合的场景

- **研究文献综述**：收集论文，综合发现
- **主题深入**：随时间建立全面的理解
- **知识综合**：跨来源连接想法
- **内容创作**：从研究生成报告、演讲、文章
- **个人学习**：在某个领域建立专业知识

### 不太适合的场景

- **简单笔记**：使用普通的 Obsidian 笔记即可
- **协作编辑**：多人编辑 wiki 违背了这个方法的目的
- **实时信息**：这个方法适用于深度、持久的知识
